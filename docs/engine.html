the engine

This is what makes vanilla.waves more than a maths library. Where p5.waves hands you numbers to draw on a canvas, vanilla.waves adds a tiny DOM engine - one shared loop that turns wave values into plain HTML. It's the signature piece: no p5, no <canvas>, cheap enough to run dozens of live elements on one page. This page is the deep-dive; the guide is the quick reference.

The idea: one number, any property

A wave is a function: pass a number in, get a number back. p5.waves feeds that number to a canvas. vanilla.waves feeds it to the DOM - a bar's scaleY, a cell's opacity, a character index, an SVG point. The engine's whole job is to call your mapping function every frame, for every marked element, as efficiently as possible.

You describe an element type once. Then any number of elements marked data-wv="<type>" come alive - driven by the same loop, each with its own sampler and its own phase.

One registered ascii type, one sampler, redrawn each frame into a single <pre>.

One shared loop, capped at 30 fps

Every marked element does not get its own requestAnimationFrame. That would multiply the browser's work by the number of elements. Instead there is exactly one loop; it walks a Map of registered elements and calls each one's update. A DOM element doesn't need 60 fps, so the loop caps itself at 30 - half the work, no visible difference for this kind of motion.

// conceptually, the shared frame:
function frame(now) {
  requestAnimationFrame(frame);
  if (now - lastFrame < FRAME_MS) return;   // 30 fps cap
  const dt = Math.min(now - prevNow, 100);   // no jump after a tab switch
  items.forEach(it => {
    if (!it.visible || !it.update) return;    // skip paused elements
    it.t += (dt / 600) * it.speed;            // each element owns its clock
    it.update(it.state, it.t);
  });
}

The loop only runs while at least one element is registered, and stops itself when the last one is destroyed. Time is per-element: each starts at its own offset (derived from its seed) and advances by data-speed, so a page full of the same type never marches in lockstep.

Pause what you can't see

An IntersectionObserver watches every element. When one scrolls out of view (plus a 120 px margin), its visible flag flips and the loop skips it - no samples, no DOM writes. Scroll it back and it resumes. That's what makes "dozens of live elements on one page" honest: only the ones on screen cost anything.

try it The tone field below is only being sampled while it's in your viewport. Scroll it off and its work stops; the shared loop keeps serving whatever is still visible.

Respect reduced motion

If the visitor's system asks for prefers-reduced-motion: reduce, the engine never starts the loop. It renders one static frame so the element still looks intentional - a shape, not a blank box - then leaves it be. The preference is watched live: flip it off and the loop starts, flip it on and everything freezes on a fresh static frame. You get the design without the movement, for free, no extra code in your renderer.

// on init, when reduced motion is requested:
if (REDUCED) items.forEach(it => { it.t += 0.6; it.update(it.state, it.t); });
else startLoop();

The lifecycle: register → init → destroy

Three calls, on the same VanillaWaves global the maths lives on.

  • register(name, def) - teach the engine an element type. def.create(node, opts, h) builds the DOM and returns a state object; def.update(state, t) mutates it each frame (optional - leave it out for a static element).

  • init(target?) - scan for [data-wv] and wire them up. No argument does the whole document; pass a selector, element, or NodeList to wire a subtree. It auto-runs once on DOMContentLoaded, and safely skips anything already wired, so calling it again after you add nodes is free.

  • destroy(target?) - stop and clean up. The element leaves the loop and the observer, and the engine removes its own attributes again. By default it keeps its last frame; give the type an optional destroy(state, node) hook if you want the DOM that create built removed too. Perfect for a loader once the real content has arrived.

// a loader that stops itself when the page is ready:
VanillaWaves.init('#boot-loader');
window.addEventListener('load', () => {
  VanillaWaves.destroy('#boot-loader');   // element, selector, NodeList, or nothing = all
});

Register types before the auto-init fires (i.e. in a script that runs during parse). Add elements or types later? Just call init() again.

The helpers do the boring parts

Your create receives a small helper kit, so a renderer stays about the mapping - not about plumbing.

HelperDoes
makeSampler(extra?)A sampler with element-friendly defaults - shift on, gentle group, low frequency, seeded from the element's data-seed. Pass extra to override (e.g. { range: [0,1] }).
norm(v)Map a wave value (≈[−1,1]) to [0,1], clamped - handy for opacity, scale, colour stops.
num(v, default)Read a data-* value as a number with a fallback.
el(tag, class?)Create an element (optionally classed) - the one-liner every renderer needs.
WThe library itself, if you want W.createSampler(...) with your own options instead of the element defaults.

Every data-* attribute on the element arrives in opts (camel-cased): data-count → opts.count, data-text → opts.text. That's how one registered type serves many configurations from markup alone.

Build an element, start to finish

Here is the entire eq renderer - the bars on the showcase. Twenty lines, and it scales from one bar to a hundred, on one shared loop, paused when offscreen.

VanillaWaves.register('eq', {
  create(node, opts, h) {
    const n = h.num(opts.count, 18);          // data-count, default 18
    const bars = [];
    for (let i = 0; i < n; i++)
      bars.push(node.appendChild(h.el('span')));
    return { bars, n, s: h.makeSampler() };    // state handed to update()
  },
  update(st, t) {
    for (let i = 0; i < st.n; i++) {
      const v = 0.08 + st.s.sample(i * 0.55, t) * 0.92;   // wave → height, sample ≈ [-1,1]
      st.bars[i].style.transform =
        'scaleY(' + Math.max(0.04, Math.min(v, 1.4)).toFixed(3) + ')';   // clamp: nooit negatief
    }
  }
});

<div data-wv="eq" data-seed="13" data-count="28"></div>

The core stays pure

The engine augments the VanillaWaves global with register / init / destroy, but it never touches the maths. waves-core.js knows nothing about the DOM; engine.js knows nothing about specific elements. And concrete components - loaders, controls, draggables - live one layer up again, in vanilla.waves_elements, importing this engine.

waves-core.js   // the maths - zero DOM, byte-identical to p5.waves
engine.js       // this: shared loop + register/init/destroy + helpers
vanilla.waves.js// the two, bundled (the CDN artifact)
                // ── elements import the engine; the engine never imports them ──

Want just the maths, no engine? Load waves-core.js alone and drive your own loop - see the starters. Want the whole thing in one tag? That's the CDN build. Either way, the numbers are the same numbers as p5.waves.
