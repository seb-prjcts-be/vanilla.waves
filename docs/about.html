about

Pass a number in, get a number back. Behind that one promise lives a curated set of 34 wave shapes: smooth sines and sharp sawteeth, gentle bumps and noise-flecked chaos, each a tiny formula tuned to drive motion, colour, shape, or pattern. vanilla.waves is that set — with nothing else loaded, and nothing drawn on a canvas.

The math never needed p5

vanilla.waves is the plain-DOM sibling of p5.waves. Look at the p5.waves source: wave(), createSampler(), the 34 formulas, the seeded shift/morph engine — all of it is pure Math.*. The only p5 touchpoint is an 8-line prototype hook at the very bottom of the file.

So this port is that canonical core, byte-for-byte, with the p5 hook removed and its own global exposed. You get the full wave dialect — the same breathing, shifting, morphing motion — to drive loaders, backgrounds, text fields, meters, any DOM element, with zero dependencies.

lineage p5.waves → p5 / canvas  ·  processing.waves → Java / Processing  ·  vanilla.waves → plain DOM, zero deps. Same dialect, different runtime — processing.waves is the precedent for "same language, maintained apart."

The challenge of a faithful port

Most ports rename and retype. This one had an easier and a harder job at once: the source language is JavaScript, so the math could be lifted verbatim — but that also means any drift from the canon is a bug, not a translation choice. The discipline:

  • Byte-for-byte core. waves-core.js is the canonical p5.waves.js math with only the 8 p5-prototype lines removed. No re-implementation, no "cleanup" — the numbers stay identical because the code is identical.

  • Own global, drop-in name. The library exposes VanillaWaves (alias vWaves), and answers to Waves as a drop-in when p5.waves isn't present.

  • Faithful entropy. shift mixes in a per-session random offset, so the same seed yields a different sequence each page load — exactly as p5.waves does. No determinism mode; the per-load variation is the point.

Trust, then verify

To prove the port never drifts, the repo runs a parity harness: it evaluates the JS reference and the ported core over the same inputs and asserts equality.

34 shapes, three temperaments

Every wave has a character. The library splits them into three groups so you can lean into the one you want:

  • Gentle: pure functions of x. Deterministic, repeatable, made of sines, cosines and modulos. The dependable section.

  • Harsh: noise-driven or tangent-spiked. They pull from a seeded PRNG, so they jitter and surprise within the wave's envelope.

  • Closing: a pool whose period divides 2π·n cleanly. A shape swept over a full cycle joins back to itself without a seam — what you want for a ring.

You don't have to pick by hand. Ask for "a gentle one" and let a seed decide, ask for a name, ask for two names and crossfade with mix, or flip shift on and watch the library cycle formulas at a tempo you set. All from one call.

How it differs from its parent

  • No canvas, no p5. There is no drawing surface. The math returns numbers; you map them onto DOM — a bar height, a character, an opacity, an SVG point.

  • A tiny DOM engine included. Register an element type, mark it with data-wv, and one shared 30 fps loop animates it. Offscreen elements pause via IntersectionObserver; prefers-reduced-motion renders one static frame instead of animating.

  • Core stays pure. The engine augments the same global but never touches the math. Element libraries import the core; the core never imports them.

Where the components live

This repo is the library — the math plus the engine. Concrete components (loaders, controls, draggables) live in a separate package, vanilla.waves_elements, built on this engine. First bib, then plugin.
