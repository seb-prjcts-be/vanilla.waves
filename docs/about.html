About vanilla.waves

One wave dialect, three runtimes. This is the one that needs nothing — no p5, no canvas, no Java, no build step. Just the math and the DOM.

the binary field — now in checkboxes—

The two-sampler binary field is where the dialect began. p5.waves draws it in pixels; here it toggles a grid of real checkboxes.

One Dialect, Many Runtimes

The "wave dialect" is a small, opinionated idea: 34 curated wave formulas, each reachable by one function call that takes a number and returns a number. Pass position in, get a value out — for size, motion, colour, density, anything. Add shift and it morphs between formulas on its own; add group: 'closing' and it does so without ever tearing a seam.

That dialect now lives in three places, each a faithful port of the same math, maintained separately:

LibraryRuntimeDraws on
p5.wavesp5.js / JavaScripta canvas
processing.wavesProcessing / Javaa PGraphics sketch
vanilla.wavesplain JavaScriptthe DOM itself

Same hero line across all three: 34 wave shapes, one function call, pass a number in, get a number back. What changes is only the runtime vocabulary — p5.waves loads two script tags, processing.waves drops a jar in your sketchbook, vanilla.waves needs a single tag and an element.

Why a DOM Port?

A canvas is one opaque rectangle. Sometimes you want the wave to live in things you already have: a real <input>, a <progress> bar, a grid of checkboxes, a line of text — accessible, inspectable, stylable with CSS, and cheap enough to run dozens of instances on one page without spinning up a renderer. That's the niche vanilla.waves fills: wave-driven interface, not wave-drawn imagery.

What's Inside

One file, two parts, zero dependencies.

PartRole
the mathThe sampler, the 34 wave formulas, the gentle/harsh/closing groups, shift/mix/seed. A byte-identical port of the canonical p5.waves core — verified 1009/1009 against v3.4.0.
the engineOne shared 30 fps requestAnimationFrame loop, IntersectionObserver pause for offscreen elements, reduced-motion support, and a register / init / destroy plugin API.

The math owns the numbers — it does not reach for a p5.waves script at runtime. That's the line between a real library and a plugin: this is the library, and it carries its own math.

The engine knows no components. Concrete element types — loaders, meters, fields — are not baked into the core. They live in the demo catalog (docs/examples.js) and, soon, in a separate vanilla.waves_elements CDN built on top of this engine. The core stays pure; elements import it, never the other way round.

Staying in Sync

Because it's a port, the one rule is: don't drift. The canonical baseline is p5.waves v3.4.0 (34 waves), pinned in docs/waves.feature.md, and a weekly drift-watch compares the language ports against the canon. A divergence in the numbers is a bug, not a feature — processing.waves set that precedent and vanilla.waves follows it.
